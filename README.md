# Music streaming SQL analysis

A small SQL project on a fake music streaming dataset. I built this to practice SQL for analyst roles I'm applying to.

The dataset is generated by a Python script in this repo. Roughly 1,500 users, 25 artists, around 200 songs and 280,000 play events. I made it up rather than using real Spotify data because that data is hard to get and I wanted something I could explain end to end.

## What's in here

Seven queries that go from easy to harder:

1. Top 10 most-played artists - basic JOIN + GROUP BY
2. Skip rate by genre - which genres do people skip the most
3. Avg plays per user, by country and plan type
4. When do people listen - plays grouped by hour of the day
5. Top song per artist - uses a window function (RANK)
6. Power users - find the top 10% of listeners and look at where they're from
7. Monthly play trend with month-over-month change - uses LAG window function

The first few are pretty basic. From query 5 onwards I'm using CTEs, window functions, and a couple of subqueries.

## Running it

```bash
python3 run_analysis.py
```

Sets up the database, loads the CSVs, and runs every query. To run just one:

```bash
python3 run_analysis.py --query 5
```

Only needs Python's standard library and SQLite (which comes with Python). Written for SQLite, but most of the queries should work on Postgres with one tweak (`strftime('%Y-%m', ...)` becomes `to_char(..., 'YYYY-MM')`).

## A few things that came out of it

- Skip rate is around 30% across every genre, which surprised me - I thought there'd be more variation. Probably an artifact of how I built the data (I made the skip probability uniform).
- The top artists by plays don't perfectly match my "monthly listeners" weights in the data generator, because user genre preferences pull plays around a bit. Bad Bunny ended up #1 because he was the only Reggaeton artist and people who liked Reggaeton concentrated on him.
- US users dominate the power-user group, which makes sense - they're the biggest country segment.
- The monthly trend is basically flat. If I'd built the data with seasonality (e.g. more plays in winter) you'd see it here. Could be a fun thing to add later.

## How this connects to analyst work

The kind of SQL in this project (group by, window functions, CTEs, ratios, time-based aggregates) is most of what analysts use day to day, regardless of industry. Skip rate, monthly trend, and power-user analysis aren't far from churn analysis, customer segmentation and KPI tracking.
